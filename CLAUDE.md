## Behavioral Guidelines (Karpathy Rules)

> Rules to reduce common errors in code assistance. If there is a conflict, the operative instructions above take precedence.

### 1. Think Before Writing Code

**Do not assume. Do not hide confusion. Surface trade-offs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — do not choose silently.
- If a simpler approach exists, say so. Push back when justified.
- If something is unclear, stop. Name what is unclear. Ask.

### 2. Simplicity First

**The minimum code that solves the problem. Nothing speculative.**

- No features beyond what was requested.
- No abstractions for code used only once.
- No unrequested "flexibility" or "configurability".
- No error handling for impossible scenarios.
- If you write 200 lines and 50 would do, rewrite it.

Test: "Would a senior engineer call this over-engineered?" If yes, simplify.

### 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When modifying existing code:
- Do not "improve" adjacent code, comments, or formatting.
- Do not refactor things that are not broken.
- Keep the existing style, even if you would do it differently.
- If you notice unrelated dead code, mention it — do not remove it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Do not remove pre-existing dead code unless requested.

Test: every modified line must trace directly back to the user's request.

### 4. Goal-Driven Execution

**Define success criteria. Iterate until verified.**

Turn tasks into verifiable objectives:
- "Add validation" → "Write tests for invalid input, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

### 5. Search Before Implementing

**Never reimplement something that already exists.**

Before writing any new function:
1. Search the codebase with Graphify (`/graphify`) or Grep to check if equivalent functionality already exists.
2. If it exists: reuse or extend. Do not duplicate.
3. If something similar but not identical exists: mention it explicitly and propose whether to extend or create new.
